# Decision Log

This is an append-only log of key architectural, product, and design decisions and their rationale. It provides historical context for how the project evolved — specs show the current state, this log shows *why*.

**Agent rule:** Do NOT write directly to this file during a session. Instead, write new entries to `decisions-pending.md` using the format below. The `scripts/merge-decisions.sh` pre-commit hook merges all pending entries into this file at commit time, then deletes `decisions-pending.md`. This keeps diffs clean and avoids churn on a shared file during multi-step sessions.

## Entry Format

```
## YYYY-MM-DD — [Decision Title]
**Decision**: [What was decided]
**Rationale**: [Why]
**Alternatives considered**: [What else was evaluated]
```

## 2026-04-19 — Per-migration version bump (not whole-run atomic)
**Decision**: `pdeqVersion` is bumped after each individual migration completes cleanly, rather than deferred until every pending migration in a batch succeeds.
**Rationale**: Directly realizes the design's observable failure-output behavior ("Recorded pdeq version: 0.3.0 — advanced from 0.2.1, 0.3.0 applied cleanly"), makes partial runs trivially resumable (re-running `/migrate` naturally picks up from the new recorded version, no special resume flag needed), and avoids any rollback machinery. The only cost is that `pdeq.json` is rewritten N times across an N-migration batch, which is harmless and arguably helpful (partial application is visible in `git status`).
**Alternatives considered**: Whole-run atomic bump — defer version writes until all migrations in a batch succeed. Rejected because failure mid-batch either requires rolling back already-applied mechanical/semantic edits (complex, error-prone) or leaves the recorded version lagging behind the actual filesystem state (violates the design's explicit "recorded version advanced to last fully-applied migration" output).
## 2026-04-19 — Scope enforcement via author discipline + post-run audit, not filesystem sandbox
**Decision**: Migration scope is enforced by (a) a required `scope:` frontmatter field that authors declare up front and (b) a post-run `audit-scope` subcommand that diffs `git status` against the declared scope and fails if writes landed outside. No sandboxed filesystem, no write-interception shim.
**Rationale**: A real sandbox (container, chroot, LD_PRELOAD write interceptor) is heavy, platform-specific, and fundamentally incompatible with the hybrid shell-plus-Claude execution model — the semantic block runs in Claude's context and cannot be sandboxed the same way as a shell block, so a sandboxed mechanical + unsandboxed semantic split would be worse than a uniform honor-system policy. The trust model (migration shell code is checked into the pdeq repo by maintainers) already assumes trust; a post-run audit catches the realistic failure mode (author forgets to declare broader scope), and the design's `scope:` frontmatter field makes violations reviewable at PR time.
**Alternatives considered**: (1) Filesystem sandbox — rejected as disproportionate for v1 and incompatible with the semantic block. (2) No scope enforcement at all — rejected because product requires `FR-migrations-scoped-writes` and `AC-migrations-scope-respected` as observable properties.
## 2026-04-19 — `/migrate` orchestration lives in markdown (Claude-driven), not pure shell
**Decision**: The `/migrate` runner is split: a markdown slash-command file (`.claude/commands/migrate.md`) orchestrates the per-migration loop in Claude's execution context, delegating deterministic work (version math, file discovery, frontmatter parsing, idempotent mechanical edits, version bumps) to subcommands of `scripts/migrate.sh`. The shell script alone is not the runner.
**Rationale**: Semantic blocks fundamentally require a live AI agent to read file context and produce judgment-driven edits. A pure shell orchestrator cannot hand control back to Claude mid-run. Conversely, shell is the right tool for version math, JSON field rewrites, and filesystem discovery — fast, testable, no runtime dependencies. Splitting the runner along this boundary keeps each half in its natural lane, matches the convention already established by `/kickoff` and `/bootstrap` (markdown orchestrator plus shell helpers), and gives QA a shell-testable surface for the deterministic half.
**Alternatives considered**: (1) Pure shell runner that shells out to `claude` CLI for semantic blocks — rejected because it couples the runner to a specific CLI invocation path, complicates testing, and breaks the "runner is a slash command" mental model. (2) Pure Claude-driven runner with no shell helpers — rejected because version math and JSON edits need to be deterministic and shell-testable without invoking an agent.
## 2026-04-19 — Bootstrap chain pins pdeq to N-1 (not N or N-2)
**Decision**: The pdeq repository adds itself as a submodule pinned to the previous stable release (N-1). On each release, the maintainer bumps the self-pin from N-1 to the newly-previous N-1 (i.e., the version just before the one being released) and runs `/migrate` against pdeq's own specs.
**Rationale**: Pinning to the in-development version (N) defeats the dogfood property — framework edits under active development would immediately take effect for the maintainer's own kickoff and status runs, making it impossible to test that the framework evolves safely under its own tooling. N-2 introduces unnecessary lag with no additional guarantee. N-1 is the minimum distance that preserves "pdeq is managed by a stable pdeq" while keeping the lag short enough that the maintainer exercises the latest stable behavior on their own repo.
**Alternatives considered**: (1) N (self-pin to in-development version) — rejected as described. (2) N-2 — rejected as unnecessary lag with no benefit. (3) Unpinned (track main) — rejected because a moving pin breaks the reproducibility that migrations are supposed to provide.
## 2026-04-19 — Breaking-change gate runs in `commit-msg`, not `pre-commit`
**Decision**: The `scripts/audit-migrations.sh` hook is invoked from the `commit-msg` git hook, not from `pre-commit`, so it can read the finalized commit message and detect the `pdeq-migration: none-required` trailer deterministically.
**Rationale**: The trailer override (design Surface 8 option C) requires reading the commit message. In `pre-commit`, the message only exists in `COMMIT_EDITMSG` during interactive commits; `git commit -m "..."` bypasses that path. `commit-msg` runs after the message is finalized and receives its path as `$1`, making trailer detection universal. The other pdeq hooks (`audit-traceability.sh`, `merge-decisions.sh`) continue to run in `pre-commit` where they belong.
**Alternatives considered**: (1) `pre-commit` reading `COMMIT_EDITMSG` — rejected as fragile with `-m`. (2) Check trailer only in CI, not locally — rejected because the gate needs to block at commit time, not after push.
## 2026-04-19 — Semver comparison uses `sort -V`, not a hand-rolled comparator
**Decision**: `scripts/migrate.sh` uses `sort -V` as the semver comparator. Two-value comparison is implemented by piping the two versions through `sort -V` and checking which came first.
**Rationale**: `sort -V` is POSIX-available on both macOS and Linux, correctly orders plain MAJOR.MINOR.PATCH including numeric-not-lexicographic segment handling (`0.3.0 < 0.10.0`), and keeps the script dependency-free. A hand-rolled comparator is more code and more edge cases to test for no benefit.
**Alternatives considered**: Hand-rolled `IFS=.` comparator — rejected for complexity. `semver` npm package — rejected because it requires node, breaking the zero-runtime-dependency constraint.
**Limitation**: `sort -V` does not correctly order pre-release suffixes (`0.4.0-rc.1`). Pre-release versions are out of scope for v1 and flagged in the engineering spec's Open Technical Questions if pdeq later ships RCs.
## 2026-04-23 — Requirement↔code mapping uses hybrid inline-marker + Code Map + audit
**Decision**: Requirement-to-code traceability is enforced by three layers: (1) inline comment markers in code citing requirement slugs at the implementation site, (2) a Code Map section in each platform's engineering spec listing planned code locations per slug, and (3) an extension of `scripts/audit-traceability.sh` that scans for markers, reconciles against product slugs and planned paths, and populates a code-locations column in `index.md`. Inline markers are authoritative; the Code Map captures intent; the audit reconciles both.
**Rationale**: Each layer catches the others' drift. Inline markers can't drift from the code they annotate (they move with the block), but they depend on author discipline — the audit catches forgotten or misspelled markers. The Code Map captures planning-time intent before code exists and surfaces stale paths when files are deleted or renamed, but would rot without an audit to force review. The audit alone (grep heuristic with no markers) would return arbitrary source lines containing slug-like strings — unreliable. Together they give agents a single lookup (`index.md` code column) that is automatically kept current by the pre-commit pipeline.
**Alternatives considered**: (1) Pure audit-driven (heuristic grep for "related" code) — rejected because no heuristic is reliable enough for agents to trust without verification. (2) Pure engineering-spec mapping (Code Map only, no markers) — rejected because mappings rot fast when code moves and there is no forcing function to update them at the moment of change. (3) Separate mapping file (e.g., `code-map.json` at repo root) — rejected because it duplicates information that already belongs with the code and the engineering spec, and introduces a third file that can drift. The hybrid approach is strictly redundant by design.
## 2026-04-23 — Slug retirement forces same-commit marker cleanup (no grace)
**Decision**: When a product-spec change removes a slug that still has markers in the code, the traceability audit blocks the commit immediately. There is no grace period for retirement — the slug-removal change and the marker-cleanup change must land together.
**Rationale**: Retirement is a deliberate authorial act, unlike the "spec written before implementation" case the grace period exists for. The author already has the whole picture at the moment they're removing the slug, so forcing the code cleanup into the same change set keeps the markdown-to-code invariant true at every commit and eliminates a class of "which repo state is real" ambiguity for downstream agents. A grace-on-retirement policy would require tracking "slug is being retired as of commit X, block after N more commits" state outside the repo — expensive and prone to bugs for a scenario that should be rare.
**Alternatives considered**: Grace-on-retirement (warn for one release, then block). Rejected — adds state, adds a slow-leak failure mode (forgotten cleanup), and defers work the author is best positioned to do now.
## 2026-04-23 — Grace period measured in commits since slug introduction, against product spec's git history
**Decision**: For the coverage grace period on newly-introduced slugs, engineering will use the number of commits that have modified the product spec file since the slug's introductory commit as the threshold signal (concrete count to be pinned in the engineering spec). Measurement is taken from `git log` on the product spec file, not from wall-clock time or release tags.
**Rationale**: Git-native, deterministic, survives clones, needs no config file or external clock. Correlates with author activity (commits = real work on the spec) rather than calendar time (which penalizes slow projects and rewards fast ones artificially). Measuring commits on the product spec file specifically, rather than global repo commits, ignores unrelated work — the grace window reflects "how much spec work has happened since this FR was added," which is the meaningful signal.
**Alternatives considered**: (1) Wall-clock days — rejected because it's not reproducible and doesn't scale across active/dormant projects. (2) pdeq release tags — rejected because most consumer repos don't tag on pdeq release cadence. (3) Global commit count — rejected because unrelated commits would burn down the grace window for reasons orthogonal to the spec.
## 2026-04-23 — Hook installation uses `git config core.hooksPath` pointing at tracked hook dir
**Decision**: `scripts/init.sh` and the `/bootstrap` command set `core.hooksPath` to a tracked directory inside the pdeq install (e.g., `.pdeq/hooks/` for consumer installs, `scripts/hooks/` for the pdeq repo self-host). The directory contains executable `pre-commit`, `commit-msg`, etc. stubs that delegate to `scripts/audit-*.sh` and `scripts/merge-decisions.sh`. A `--skip-hooks` flag on `init.sh` and an equivalent prompt in `/bootstrap` lets CI/container contexts opt out.
**Rationale**: `core.hooksPath` is a single git config, survives clones across all developers by virtue of the install step that runs once per clone, and keeps the hook scripts themselves tracked in the repo — so adding or changing a hook is a code review, not a per-developer re-install. This matches pdeq's tracked-deterministic-zero-runtime-deps philosophy and aligns with how `/kickoff` and `/migrate` already work. Symlink-into-`.git/hooks/` was considered but rejected as drift-prone: nothing forces a re-install after new hooks are added. Third-party frameworks (pre-commit.com) were rejected as introducing a Python dependency and a parallel tool consumers must learn.
**Alternatives considered**: (1) `scripts/install-hooks.sh` that symlinks into `.git/hooks/` — rejected because it must be rerun after every clone and every new-hook addition, and there's no forcing function. (2) `pre-commit.com` framework — rejected as a Python dependency that conflicts with the zero-runtime-deps constraint. (3) Layered hooks (pdeq's hooks chain into a user-provided `.git/hooks-local/*`) — accepted as a non-blocking follow-up to let consumers layer their own hooks without fighting the pdeq install; not required for v1.
