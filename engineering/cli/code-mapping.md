# Requirement ↔ Code Mapping — CLI Technical Spec

> Based on requirements in `../../product/code-mapping.md`
> (No design spec — feature has no UI surface. Design lane is explicitly N/A.)

## What We're Building

Technically, this feature is three loosely-coupled additions to the existing traceability pipeline: (a) a short, regex-defined **inline marker** written into source files in the appropriate comment syntax for each language, (b) a **Code Map** table embedded in each platform's engineering spec that lists planned code locations per slug, and (c) an extension of `scripts/audit-traceability.sh` that scans the repo for markers, reconciles them against product-defined slugs and Code Map entries, and rewrites a **Code** column in `index.md`. No new script, no new config file, no new runtime dependency — just new phases in the existing pre-commit audit.

The design is shaped by two decisions. First, **the regex is canonical, the language table is advisory**: a single extension-indexed table picks which comment delimiter is legal per file type, but the slug-matching grammar itself is language-agnostic and runs as one ripgrep invocation against the whole tree. Agents do not need to know per-language comment syntax to place a marker; they follow the table or copy from a neighboring file. Second, **the index is derived, not authored**: the Code column in `index.md` is regenerated by the audit on every run and never edited by hand, eliminating the write-then-forget failure mode that plagues audit columns in most traceability tools. The authoritative mapping lives in the code itself; `index.md` is a cache with a deterministic rebuild path.

## Technical Approach

### Marker grammar — one regex, language-aware anchoring

Every marker matches this single grammar, regardless of file type:

```
<comment-open> *Implements: *<slug>(?: *, *<slug>)* *<comment-close>
```

where:
- `<slug>` is `(FR|NFR|AC)-[a-z0-9-]+(?![a-z0-9-])` — the trailing negative lookahead is mandatory and prevents a shorter slug prefix from matching inside a longer slug (e.g. `FR-ex-auth` from matching inside `FR-ex-auth-login`). This is what makes `AC-code-mapping-near-match-rejected` hold.
- `<comment-open>` and `<comment-close>` are drawn from a per-extension table (see §Marker syntax reference).
- `<comment-close>` is the literal close token (e.g. `-->`) for kinds that require one (HTML, Markdown, CSS). For kinds where it is absent (C-family, shell, SQL), the regex fragment is the empty string.

The full marker must appear on a single source line. For kinds whose comment form requires a close token, the close token must appear on the same line as the open — a marker whose `<!--` has no matching `-->` on the same line is ignored (neither counted nor reported as an orphan). This is what guarantees markers are grepped atomically, one per line.

Line-anchoring is realized by the `^.*<comment-open> *Implements:` prefix of the scan: the audit only counts matches where the comment-open token and the `Implements:` literal are on the same matched line, satisfying `FR-code-mapping-marker-language`.

### Known precision limitations

Two cases that the regex cannot distinguish without a full language parser, documented as accepted limitations:

- **Nested comments.** An `// Implements: FR-x` that lies *inside* a `/* … */` block comment is matched by the regex even though the outer block comments it out. Parsing block-comment nesting requires a real tokenizer. Accepted: authors who comment out a marker entirely should delete it; the audit will otherwise treat it as live. `NFR-code-mapping-precision` is therefore "zero false positives on near-matches", not "zero false positives in all pathological cases."
- **Strings that look like markers.** A string literal `"// Implements: FR-x"` embedded in source code is matched. Accepted: this is exceedingly rare and easy to diagnose when it occurs.

These limitations are documented in `engineering/CLAUDE.md` so future agents know to avoid the antipatterns rather than expect the audit to catch them.

### Marker syntax reference

The audit uses this table, indexed by file extension, to decide which comment-open token is legal per file. Each entry is the regex fragment plugged into the marker grammar. An `Implements:` literal that appears in a file but does not match the table entry for its extension is not counted as a marker (neither positively nor as an orphan), which satisfies `AC-code-mapping-marker-syntax-per-type`.

| File kinds | Extensions | Comment open | Comment close | Example |
|---|---|---|---|---|
| C-family | `.ts, .tsx, .js, .jsx, .mjs, .cjs, .go, .swift, .java, .kt, .c, .cc, .cpp, .h, .hpp, .rs, .scala, .dart` | `//` | — | `// Implements: FR-auth-email-login` |
| Shell / scripting | `.sh, .bash, .zsh, .py, .rb, .pl, .yaml, .yml, .toml` | `#` | — | `# Implements: FR-migrations-version-bump` |
| SQL | `.sql` | `--` | — | `-- Implements: FR-db-read-replica` |
| HTML / Markdown | `.md, .html, .xml, .svg` | `<!--` | `-->` | `<!-- Implements: FR-kickoff-roadmap-pull -->` |
| Block-comment only | `.css, .scss` | `/*` | `*/` | `/* Implements: FR-ui-dark-mode */` |

The table is maintained in `scripts/audit-traceability.sh` as a small associative array. Adding a new language is a one-line change. Extensions not in the table are excluded from the scan — unknown-extension files cannot carry markers until the table is updated.

Satisfies `FR-code-mapping-marker-language`, `FR-code-mapping-marker-presence`, `FR-code-mapping-marker-multi`, `FR-code-mapping-marker-slug-reference`.

### Scope rule — "smallest enclosing named unit"

`FR-code-mapping-marker-scope` requires that a marker sit on or immediately above the smallest enclosing named unit that implements the requirement. The audit enforces a weaker proxy via the `check_scope_rule` function (phase 5b, between marker scan and orphan check): **a marker is flagged when the file contains at least one named-unit declaration AND the marker line is more than one line before the first such declaration.** Position is judged structurally — against the first declaration's line — not against an absolute line count.

Detection of "function-like definition is possible" is by file kind:

- C-family, Python, Ruby, Swift, Go, Rust, shell (when the shell script contains at least one `function NAME` or `NAME() {` declaration) — scope rule applies.
- Markdown, HTML, YAML, JSON — scope rule does not apply (no functions).
- Shell scripts with only top-level statements — scope rule does not apply (detected by scanning the whole file for function declarations; zero found = exempt).

Algorithm (per marker, per applicable file):

1. Find `first_decl_line`: the lowest line number in the file matching the named-unit regex (`function`, `def`, `func`, `class`, `export function`, JSX/TSX component declarations, etc.).
2. If no declaration matched, the file is exempt — no warning regardless of marker line.
3. If `marker_line >= first_decl_line`, the marker sits at or below the first named unit (i.e., inside its body or below a later one). No warning.
4. If `marker_line == first_decl_line - 1`, the marker is immediately above the first named unit — the canonical correct placement. No warning.
5. Otherwise (`marker_line < first_decl_line - 1`), the marker is at file-top: it precedes every named unit in the file and is not paired with the first one. Warn (phase 5b is not a block, because structural detection is imperfect). Flagged markers do not count toward coverage. Non-flagged markers count normally.

The earlier "marker line ≤ 5 = file top" heuristic produced false positives on short components whose first declaration started on line 2 or 3 (a marker on line 3 that is semantically inside the function body would trip the warn). Anchoring against `first_decl_line` instead of an absolute line number eliminates that false-positive class without weakening the catch for "agent pasted `// Implements:` at file top and never moved it."

This is still a deliberately loose check. Precision on structural placement is impossible in a regex scanner without language-aware parsing; the goal is to catch the obvious file-top antipattern, not to validate every structural placement. Satisfies `AC-code-mapping-marker-scope-enforced`.

### Code Map section — format and parse contract

Each platform engineering spec includes a `## Code Map` H2 section. The section contains a Markdown table with exactly three columns, in this order:

| Column | Content |
|---|---|
| Slug | A single `FR-<feature>-<slug>` slug (one slug per row; multi-slug markers are a code concern, not a Code Map concern). |
| Planned location | A relative path from repo root, optionally with a line range (`scripts/migrate.sh:120-145`). Multiple locations separated by `; `. Empty string `—` means "no specific location planned." |
| Status | One of: `implemented`, `planned`, `unimplemented`. |

`implemented` means the location exists and a marker at that location cites the slug. `planned` means the location does not yet exist (or has no marker yet) but will. `unimplemented` is the escape hatch for `FR-code-mapping-acknowledged-unimplemented`: the slug is deliberately deferred; the audit does not warn or block on it regardless of grace-period status.

The audit parses this table by locating the `## Code Map` heading and reading rows until the next H2/H1 heading or EOF. Whitespace handling is spelled out: leading and trailing spaces in each cell are trimmed; runs of internal spaces are preserved as-is (they don't appear in the vocabulary we care about). Tabs are accepted as column separators only where the Markdown table syntax already allows them. The column count must be exactly three after trimming; the `Status` cell must be exactly one of `implemented`, `planned`, or `unimplemented` after trimming (case-sensitive). Any deviation is a phase-7 parse error, not a silent skip.

Satisfies `FR-code-mapping-planned-paths`, `FR-code-mapping-planned-paths-living`, `FR-code-mapping-planned-paths-per-platform`, `FR-code-mapping-acknowledged-unimplemented`, `AC-code-mapping-planned-paths-living`, `AC-code-mapping-acknowledged-unimplemented`.

### Audit algorithm

The extended `scripts/audit-traceability.sh` runs these phases in order. Phases 1–4 are the existing audit; phases 5–9 are new.

1. Parse product specs → set of defined slugs (existing).
2. Parse design/engineering/qa specs → set of referenced slugs (existing).
3. Compare sets → report orphan references (existing).
4. Validate `index.md` file paths exist (existing).
5. **Marker scan**: ripgrep the tree (excluding paths in §Exclusions) for `^.*(<comment-open-alternation>) *Implements: *(<slug-grammar>)( *, *<slug-grammar>)* *(<comment-close-alternation>)`. For each match: record `(slug, file, line)` tuples. Reject any marker citing a slug not in the product-defined set → `AC-code-mapping-orphan-marker-rejected`.
   - **Phase 5b (`check_scope_rule`)**: per §Scope rule, for each counted marker, locate the first named-unit declaration line in the same file. Warn iff the marker line is more than one line before that declaration line (i.e., not on or immediately above any named unit). Flagged markers do not count toward coverage. Warn only, no block. → `AC-code-mapping-marker-scope-enforced`.
6. **Retirement check**: for each marker in the tree, if the cited slug is in the current product-defined set: OK. If not: block, naming the retired slug and all files that still cite it → `AC-code-mapping-retirement-blocks`. (This is a tightened form of phase 5 — it runs even when no new markers were added in the current commit.)
7. **Code Map validation**: for each platform engineering spec, parse Code Map rows. Per-row checks:
   - For each row with `Status: implemented`: verify every `planned_location` file exists AND a marker citing the slug exists somewhere in the tree. Either a missing file or the absence of a marker is a block → `AC-code-mapping-stale-planned-path-rejected`, plus the implemented-but-no-marker check that catches "author renamed the code but forgot to update the Code Map."
   - For each row with `Status: planned`: the location is aspirational — file may not exist yet. No path or marker check at this phase. Phase 8 handles coverage, which is where missing markers for planned slugs get caught after the grace period.
   - For each row with `Status: unimplemented`: record the slug as exempt from phase 8. No further checks.

   Note: the product AC `AC-code-mapping-stale-planned-path-rejected` uses the word "references" loosely. Engineering's Status vocabulary interprets it as "implemented rows reference a real file." A planned row with a non-existent file is a statement of intent, not a reference — the grace period + phase 8 is the forcing function that pushes a planned row toward `implemented` once the code lands.
8. **Coverage check**: for each `FR-` slug in the product-defined set, check whether at least one marker cites it OR it is Code-Map-`unimplemented`. For each uncovered FR, compute grace status (see §Grace period). Uncovered-within-grace: warn. Uncovered-past-grace: block. → `FR-code-mapping-audit-coverage`, `FR-code-mapping-audit-coverage-blocks`, `FR-code-mapping-audit-coverage-grace`, `AC-code-mapping-uncovered-warns`, `AC-code-mapping-uncovered-blocks`.
9. **Index rewrite**: regenerate the `Code` column for every slug that has markers. Two modes:
   - **Default (pre-commit hook context)**: rewrite `index.md` in place and re-stage it so the commit includes the updated column. This realizes "index entries never need manual editing" from `FR-code-mapping-index-populated`. The commit message trailer `pdeq-audit: index-code-column-updated` is appended when this happens, making the edit visible in `git log`.
   - **Strict (CI, manual invocation)**: the `--check` flag inverts the behavior — if rewriting would produce a diff, exit 1 with "index.md Code column out of date — run ./scripts/audit-traceability.sh locally and commit the result." Used in CI to detect if someone committed without running the pre-commit hook (e.g., `--no-verify`).
   - `PDEQ_CODE_MAPPING_SKIP_INDEX_REWRITE=1` short-circuits the phase in both modes (used internally when the pre-commit hook runs the audit twice in quick succession).
   - Per-platform scoping: each marker's file path already encodes its platform via `engineering/apps/<platform>/…`; the Code column lists all markers as `<path>:<line>` entries. Merging or splitting by platform is not needed because the path prefix makes per-platform scoping readable at a glance. → `FR-code-mapping-index-populated`, `FR-code-mapping-index-code-locations`, `FR-code-mapping-index-removes-stale`, `AC-code-mapping-index-reflects-markers`, `AC-code-mapping-index-drops-removed`, `FR-code-mapping-planned-paths-per-platform`.

Phases 5–9 form a single ripgrep pass for the marker scan plus in-memory reconciliation, keeping the total audit under the 2s target on the pdeq repo (`NFR-code-mapping-audit-speed`, `AC-code-mapping-audit-speed`).

### Grace period — commit count against the product spec file

The grace period is **5 commits that modified the containing product spec file since the slug's introductory commit**. Measurement algorithm:

1. For each uncovered FR slug `S`, run `git log --diff-filter=A --format=%H -S"$S" -- product/<feature>.md` and take the most recent commit — this is `intro_commit(S)`. The `--diff-filter=A` ensures we only match the commit that *added* the slug, not any later commit that touched nearby text containing `S`. If the same slug was added, removed, and re-added (rare; violates the "slugs are permanent" rule anyway), the most recent addition wins.
2. Count `git rev-list --count HEAD -- product/<feature>.md` entries since `intro_commit(S)`: that is `delta(S)`.
3. If `delta(S) < 5`: warn. Else: block.

The `5` is configurable via `PDEQ_CODE_MAPPING_GRACE` (integer env var) for CI or migration scenarios; default is `5`. Rationale for 5: two commits often suffice for "spec + code" when done together, five covers "spec first, iterate, code in a follow-up PR" without letting uncovered FRs sit indefinitely. Decision recorded in `decisions-pending.md` (2026-04-23).

Edge cases:
- **Shallow clones weaken the grace guarantee.** If `intro_commit(S)` is not reachable (shallow clone cut it off), `git log` returns empty and the audit treats the slug as `delta = 0` (full grace) with a warning printed. CI **must** use a full clone (`fetch-depth: 0` on GitHub Actions) — a shallow CI clone would silently let uncovered FRs through indefinitely. The CI pipeline documentation notes this requirement.
- **Rebased / squashed history** — `delta` is measured against current `HEAD`'s view of history, so rebases that combine commits may shorten the grace window. Acceptable — the author is actively working on the spec in either case.
- **First-introduction commit not yet made** — if the slug is being added in the commit currently being scanned, `intro_commit(S)` is that commit itself and `delta(S) = 0`. The grace is full, as intended.

Satisfies `FR-code-mapping-audit-coverage-grace`.

### Escape hatch

Reuses the existing `PDEQ_ALLOW_DRIFT=1` mechanism from the drift-detection convention. When set:
- Phase 5 orphan-marker check is demoted from block to warn.
- Phase 6 retirement check is demoted from block to warn.
- Phase 7 stale-planned-path check is demoted from block to warn.
- Phase 8 uncovered-past-grace check is demoted from block to warn.
- Phase 9 index rewrite proceeds in place (no block), with the override logged.

The audit report includes a line per suppressed condition when the override is active, naming what was suppressed. Satisfies `FR-code-mapping-audit-escape-hatch`, `AC-code-mapping-escape-hatch`.

### Exclusions

Default exclusion list (baked into `scripts/audit-traceability.sh`):

- `.git/`, `node_modules/`, `vendor/`, `dist/`, `build/`, `target/`, `.venv/`, `__pycache__/`
- Files matching `*.lock`, `*.min.js`, `*.map`
- `.pdeq/` — excluded unless `pdeq.json:selfHost` is `true`. The `selfHost` flag is set exclusively in the pdeq repository's own `pdeq.json`, so consumer repos never scan into their submodule. In the pdeq repo itself, `selfHost: true` tells the audit to scan `scripts/`, `.claude/commands/`, and the other framework files that live at the repo root. This replaces the earlier heuristic of detecting self-host via filesystem sniffing — explicit flag avoids false positives in consumer repos that happen to have a `VERSION` file.
- Any path matching `glob(pdeq.json:codeMappingExclude)` (optional array of glob patterns, defaults to `[]`). Appended to the default list; does not replace it.

Exclusions are pre-filtered via ripgrep's `--glob` flag or grep's `--exclude-dir` for the grep fallback.

## Data Model

### In-memory state during audit

- `product_slugs: Set[slug]` — union of all slugs extracted from `product/*.md` normalized content.
- `markers: List[(slug, file, line)]` — every valid marker hit, in discovery order (by file path, then line).
- `code_map: Dict[(platform, slug) → (path, status)]` — parsed Code Map rows per engineering spec.
- `uncovered_frs: Set[slug]` — FRs in `product_slugs` with no marker and no `unimplemented` Code Map row.
- `grace_deltas: Dict[slug → int]` — commit count per uncovered FR, populated lazily (only for uncovered slugs).

### On-disk state

Only `index.md` is written by the audit; all other files are read-only. Specifically, the `Code` column of each row in the Index table is regenerated. If a slug has no markers, the Code column is empty. If the slug has markers, the column lists `<path>:<line>` entries comma-separated — one per marker. **No line ranges.** The earlier draft specified `line_start-line_end` with a forward-scan heuristic for the end line; that heuristic was non-deterministic across languages and added complexity for no grep/navigation benefit (editors jump to `file:line` natively). Keeping it to `file:line` preserves determinism and matches how `grep -n` and IDE jumpers already format locations.

Example index.md row after an audit run:

```
| FR-kickoff-roadmap-pull | FR | product/kickoff.md | design/cli/kickoff.md, engineering/cli/kickoff.md, qa/cli/kickoff.md | .claude/commands/pdeq-kickoff.md:142, scripts/kickoff.sh:78 |
```

Index header gains one column: `Code`. Existing rows gain an empty last cell on first run and fill on the first audit pass after implementation.

## API / Interface Design

### `scripts/audit-traceability.sh` — new functions

| Function | Purpose |
|---|---|
| `scan_markers` | Runs the ripgrep-or-grep marker scan and emits `(slug\tfile\tline)` tuples on stdout. |
| `check_scope_rule` | Reads `scan_markers` output. For each marker in a function-capable file, finds the first named-unit declaration line and warns iff the marker is more than one line above it. Warning only. |
| `parse_code_map <engineering-spec>` | Emits `(slug\tpath\tstatus)` tuples for one engineering spec's Code Map section. |
| `check_orphan_markers` | Reads `scan_markers` output, compares against `$PRODUCT_SLUGS`, reports orphans. |
| `check_retirement` | Subset of `check_orphan_markers` specifically for markers whose slug was present in a prior product spec version but not the current. |
| `check_code_map_paths` | For each engineering spec, parses Code Map rows and verifies planned-location files exist. |
| `check_coverage` | For each FR slug, determines warn/block/pass status using grace calculation. |
| `grace_delta <slug> <product-spec>` | Computes the commit-count-since-intro for a slug. Returns integer on stdout. |
| `rewrite_index_code_column` | Rebuilds the Code column in `index.md` from marker data. Default mode rewrites in place and re-stages; `--check` mode exits 1 if a diff exists without writing. |

All functions honor the existing `audit-traceability.sh` conventions: stdout for tool-consumable output, stderr for human-readable logs, exit codes propagate.

### Environment variables

| Variable | Default | Effect |
|---|---|---|
| `PDEQ_ALLOW_DRIFT` | unset | When `1`, demotes all code-mapping-audit failures to warnings and rewrites `index.md` in place. |
| `PDEQ_CODE_MAPPING_GRACE` | `5` | Integer, overrides the commit-count threshold for the grace period. |
| `PDEQ_CODE_MAPPING_SKIP_INDEX_REWRITE` | unset | When `1`, skips phase 9 entirely (useful in pre-commit scenarios where the audit runs twice and the second run should not re-diff). |

### `pdeq.json` additions

One optional field:

```json
{
  "codeMappingExclude": ["docs/legacy/**", "scripts/third-party/**"],
  "selfHost": true
}
```

`codeMappingExclude` is appended to the default exclusion list, not replacing it. Empty or absent → defaults only. `selfHost` defaults to `false`; the pdeq repo's own `pdeq.json` sets it to `true` so the audit scans into framework files under `scripts/`, `.claude/commands/`, and `hooks/` that would otherwise be treated as submodule-internal.

### Interaction with other audits

`audit-traceability.sh` already runs multiple phases in a single invocation. ## State Management

The audit is stateless across runs. Every invocation rebuilds its in-memory state from the filesystem. No lock files, no cache files, no `.pdeq-audit-state/` directory. Two consecutive runs with no file changes produce byte-identical output, satisfying `NFR-code-mapping-determinism` and `AC-code-mapping-deterministic-output`.

The one apparent exception is the `Code` column in `index.md` — but that is an output artifact, not state. If `index.md` is deleted, the next audit run writes it from scratch (assuming a template header exists) or fails cleanly with "index.md missing."

| Ripgrep missing, grep fallback slow | 0 (warn) | `ripgrep not found; using grep fallback — audit may be slower than 2s target` | No |

All messages are on stderr. stdout carries the machine-readable `(slug, file, line)` tuples when requested via `--emit-markers` subflag (future; not in v1).

## Performance Considerations

### Target: sub-2-second audit on the pdeq repo

- Marker scan: one ripgrep invocation with `--type-add` for each kind in the syntax table. Ripgrep's parallelism handles ~10k-file repos in well under 1s.
- Grep fallback: `grep -rnE` with `--include` globs per file kind, invoked in parallel via xargs when available. ~3–5× slower than ripgrep but still sub-second on pdeq-sized repos.
- Code Map parse: O(engineering specs × rows). Pdeq has one engineering spec per platform; rows are ~10s per spec. Negligible.
- Index rewrite: O(slugs) string building. Negligible.
- Grace delta: `git log -S<slug>` is the most expensive per-slug operation (~50ms per slug on a deep repo). Mitigation: only computed for uncovered slugs, which should be a small fraction.

Measurement: the existing audit's `time` wrapper is extended to report phase-level timing on `PDEQ_AUDIT_PROFILE=1`, for debugging perf regressions.

### Determinism

All operations that could be nondeterministic are pinned:

- Ripgrep output is sorted before consumption (`rg --sort path`).
- Slug sets are stored in sorted arrays, not hash sets.
- Index rewrite emits rows in the same order they appeared before the rewrite, preserving diff locality.
- Git operations use `--no-abbrev` and explicit formats.

## Security Considerations

- The audit reads only; no command injection surface exists except where slug names feed into `grep -S` and `git log -S`. Slugs are validated against `[A-Z]+-[a-z0-9-]+` before being interpolated, eliminating metacharacter concerns.
- Excluding `.pdeq/` prevents accidentally scanning a submodule checkout that contains markers from a different namespace. The pdeq repo itself self-hosts and needs its own markers scanned; in the pdeq repo, `.pdeq/` is not an exclusion (detected via "is this the pdeq repo?" heuristic: presence of `VERSION` at repo root + `scripts/audit-traceability.sh` authored here, not via submodule).
- `pdeq.json:codeMappingExclude` patterns are passed verbatim to ripgrep/grep. They cannot escape the repo tree because the scan is rooted at repo root.

## Implementation Plan

Each step below includes a one-line rationale for ordering. Atomic commits preferred.

1. **Update `engineering/CLAUDE.md` with Code Map format.** Rationale: defines the format before anything parses it. Template only — no spec edits yet. Dependency for PDEQ-fmeluqnd.
2. **Backfill Code Map in `engineering/cli/migrations.md`.** Rationale: proves the format on a real existing spec, smokes out format issues before audit parsing lands. All migration FRs go in as `implemented` rows with real paths. Any markers added to `.md` files in this step (notably in `.claude/commands/pdeq-migrate.md`) use HTML-comment syntax `<!-- Implements: FR-… -->` — mixing with `//` or `#` in Markdown files will not be recognized by phase 5.
3. **Implement `scan_markers` + syntax table in `audit-traceability.sh` (phase 5).** Rationale: unlocks orphan detection, the highest-value guarantee. At this step, phases 6–9 are stubs.
4. **Implement `check_orphan_markers` + `check_retirement` (phases 5–6).** Rationale: first observable audit behavior. At this point the audit can reject invalid markers without yet understanding coverage.
5. **Backfill `// Implements:` markers in existing pdeq scripts/commands/hooks for every current FR.** Rationale: must happen before coverage check goes live or every existing slug blocks. Covered by PDEQ-nydghudw.
6. **Implement `parse_code_map` + `check_code_map_paths` (phase 7).** Rationale: requires the Code Map format to be stable in specs, which step 1–2 guaranteed.
7. **Implement `check_coverage` + `grace_delta` (phase 8).** Rationale: builds on orphan detection and Code Map parsing; grace is the subtlest piece, so defer until earlier phases are proven.
8. **Implement `rewrite_index_code_column` (phase 9) + add `Code` column to `index.md`.** Rationale: output-only; safe to land last because breaking the rewrite only dirties `index.md`, doesn't lose data. First run regenerates the column from the markers added in step 5.
9. **Update `/pdeq-kickoff` engineering subagent prompt.** Rationale: enforces the Code Map convention on all new features. Not a hard blocker since step 1 already documents it, but closes the loop.
10. **Update root `CLAUDE.md` with requirement↔code mapping section.** Rationale: top-level discoverability for new agents. Written last so it reflects actual implemented behavior.
11. **Write bats tests per QA test plan.** Rationale: covered by PDEQ-seloclil; slots in after step 8.

Open technical questions (resolve in review or during impl):

- **Line-range detection heuristic accuracy.** The 20-line cap on forward-scan may be too short for long functions. Not worth parsing ASTs for; acceptable to ship the heuristic and tighten if agents complain.
- **Behavior when `rg` is absent.** Grep fallback should work on all POSIX systems but has not been profiled on a 10k-file repo. If it exceeds 2s, fail the coverage phase with a `ripgrep recommended for repos > N files` message rather than silently degrading. Threshold to be set after measurement.
- **Interaction with git LFS / submodule files.** Scanning into LFS pointers or submodule contents would be expensive and meaningless. Current plan: exclude `.gitmodules` targets automatically. Verify in implementation.
