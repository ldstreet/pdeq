# Migration 0.3.0 — Rename pdeq slash commands to `pdeq-` prefix

## Context

Pdeq 0.3.0 namespaces every pdeq-installed slash command under the `pdeq-`
prefix. Pre-0.3.0, pdeq shipped six bare-verb commands (`/bootstrap`,
`/impact`, `/kickoff`, `/migrate`, `/status`, `/visualize`); 0.3.0 renames
all six to their `pdeq-` form (`/pdeq-bootstrap`, `/pdeq-impact`,
`/pdeq-kickoff`, `/pdeq-migrate`, `/pdeq-status`, `/pdeq-visualize`). The
`/pdeq-update` command, introduced earlier, was already prefixed and is
unaffected.

Why the rename: a maintainer browsing their slash-command palette had no
way to tell which commands came from pdeq versus their own project or
other tooling, and bare verbs collide with whatever bare-verb commands a
consumer's other tools may ship. Typing `/pdeq` and tab-completing now
surfaces the entire pdeq command surface at once, and ownership is
unambiguous at a glance.

What the consumer sees in their diff after `/pdeq-migrate` runs: in
`<git-root>/.claude/commands/`, every old bare-name symlink (e.g.
`bootstrap.md`) is replaced by its prefixed equivalent (`pdeq-bootstrap.md`)
pointing at the same file inside the `.pdeq/` submodule. Nothing else is
touched. If the consumer had previously customized one of these command
files (so the entry is a regular file rather than a pdeq-managed symlink),
the migration leaves the customization in place and prints a one-line
warning to stderr telling the consumer to rename it manually.

## Mechanical

```shell
# Idempotently rename pdeq command symlinks under .claude/commands/.
#
# For each pre-0.3.0 bare-verb command, this block:
#   * If <name>.md is a pdeq-managed symlink   → remove it and create
#     pdeq-<name>.md pointing at the prefixed source in the submodule
#     (only if pdeq-<name>.md does not already exist).
#   * If <name>.md is a regular file (consumer customization) → leave it
#     alone, print a one-line stderr warning naming the file.
#   * If <name>.md is absent and pdeq-<name>.md is present → no-op.
#
# Re-running this block against already-migrated state is a no-op: the
# old name is gone, the new name exists, every branch short-circuits.
#
# Touches only .claude/commands/. Does not modify anything outside that
# directory. Compatible with both consumer (where the dir is a real
# directory at <git-root>/.claude/commands/) and self-host (same path).

set -eu

COMMANDS_DIR=".claude/commands"

# Old → new pairs. The submodule path is fixed at .pdeq/.claude/commands/
# in consumer installs; init.sh symlinks always use ../../.pdeq/.claude/commands/<name>
# as the symlink target so `dirname` resolution works from inside .claude/commands/.
NAMES="bootstrap impact kickoff migrate status visualize"

if [ ! -d "$COMMANDS_DIR" ]; then
  # Nothing to migrate — consumer hasn't run init.sh yet, or the directory
  # was removed. The /pdeq-migrate runner will still bump the recorded
  # version; subsequent installs ship the prefixed names directly.
  exit 0
fi

for name in $NAMES; do
  old="$COMMANDS_DIR/${name}.md"
  new="$COMMANDS_DIR/pdeq-${name}.md"
  target="../../.pdeq/.claude/commands/pdeq-${name}.md"

  # If the new symlink already exists, we either ran before or the consumer
  # is on a fresh install. Either way, leave it alone — and if the old name
  # is also still there as a symlink, treat the new as authoritative and
  # remove the old.
  if [ -L "$new" ] || [ -e "$new" ]; then
    if [ -L "$old" ]; then
      rm "$old"
    fi
    continue
  fi

  if [ ! -e "$old" ] && [ ! -L "$old" ]; then
    # Neither old nor new present — nothing to do for this command.
    continue
  fi

  if [ -L "$old" ]; then
    # pdeq-managed symlink — replace with the prefixed form.
    rm "$old"
    ln -s "$target" "$new"
    continue
  fi

  if [ -f "$old" ]; then
    # Regular file = consumer customization. Don't touch it.
    printf '[pdeq 0.3.0] %s is a regular file (customized) — leaving alone. Rename to %s manually if you want the prefix convention.\n' \
      "$old" "$new" >&2
    continue
  fi
done
```

## Notes

The migration deliberately scopes its writes to `.claude/commands/` —
no other directory is touched. The `scripts/audit-scope` post-run check
validates this against `scope: default`, which expands to the consumer's
specsRoot plus `pdeq.json`; the `.claude/commands/` writes happen at the
git root rather than under specsRoot, so consumers with a non-`.` specsRoot
may need to re-check the audit-scope rule. The `default` scope expansion
already includes `**` for root-level installs, and for nested installs the
`.claude/commands/` directory lives at the git root (per `init.sh` Step 6),
which is captured by the default `**` glob in self-host context.

Idempotency rests on three branches: (a) new exists → skip and clean up
old; (b) neither exists → skip; (c) old exists as symlink → rename;
(d) old exists as regular file → warn and skip. Re-running after a
successful apply takes branch (a) for every command. Re-running on a
fresh install with only prefixed names takes branch (b).

The pdeq self-host case: pdeq's own repo's `.claude/commands/` already
contains the prefixed names directly (renamed in the same commit that
authored this migration). When the maintainer eventually re-pins
`.pdeq/` to 0.3.0 and runs `/pdeq-migrate` against the pdeq repo's own
specs, every iteration of the loop takes branch (b) — the old names are
gone, the new names exist as regular files (not symlinks, since pdeq
self-host doesn't symlink). Branch (a) does not fire because the old
names are absent. The block exits 0 with no changes, satisfying the
self-migration idempotency requirement.
